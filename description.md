# 校园智能学习生活助手 Agent 系统

## 项目描述

独立完成一个面向大学生的校园智能学习生活助手 Agent 系统，围绕“自然语言驱动校园服务”构建统一对话入口。用户可通过聊天完成课表查询、课程资料检索、提醒管理、天气查询、路线规划、出行准备建议等操作。项目采用前后端分离架构，后端基于 Spring Boot + LangChain4j 实现 Agent 编排、Tool 调用、RAG 检索与 MCP 外部能力接入，前端基于 Vue 3 实现流式对话交互。

## 核心技术栈

- 后端：Spring Boot 3、Java 21、Maven
- Agent / LLM：LangChain4j、LangChain4j Reactor
- 智能检索：Embedding、RAG、RedisSearch
- 数据库：MySQL、MyBatis-Plus
- 外部能力接入：MCP（高德地图 MCP、Open-Meteo MCP）
- 前端：Vue 3、Vite、Axios、Fetch Stream API、原生 CSS
- 其他：WebFlux、SSE 流式输出、会话记忆、CORS、全局异常处理

## 核心职责

### 1. 负责项目整体方案设计与从 0 到 1 搭建

- 设计校园智能助手业务边界，将课表、资料、提醒、天气、路线等能力统一抽象为对话式入口
- 设计后端分层架构，划分 Controller、Service、Agent、Tools、RAG、MCP、Repository、Config 等模块
- 搭建前后端分离项目结构，完成后端服务与前端聊天页面的整体落地

### 2. 负责 Agent 体系设计与 Prompt 工程

- 基于 LangChain4j 构建同步 Agent 与流式 Agent，支持自然语言问答与多轮对话
- 为 Agent 设计系统提示词，明确 Tool 调用优先级、RAG 使用时机和天气/路线查询约束
- 将“出行准备建议”能力抽象为 Skill 规则并落地到 Agent Prompt 中，约束其必须先查课表、再查天气、最后按规则生成建议

### 3. 负责 Tool 与业务能力封装

- 封装 `getSchedule`、`queryMaterial`、`addReminder`、`listReminders`、`getWeather`、`queryroute` 等 Tool
- 将课表查询、提醒管理、资料检索等本地业务能力统一注册到 Agent 中，实现 Agent 自动决策调用
- 设计课表时间解析、提醒抽取、自然语言查询规则，支持校园场景下的结构化能力调用

### 4. 负责 RAG 检索模块实现

- 设计课程资料检索流程，完成文档加载、切分、Embedding、向量存储、检索返回
- 基于 Redis 向量检索能力构建课程资料知识库，实现期末试卷、复习资料、课程文档等内容的相似度检索
- 对接 LangChain4j `EmbeddingStoreIngestor` 与 `ContentRetriever`，实现 Agent 检索增强问答

### 5. 负责 MCP 外部能力接入

- 接入 `@amap/amap-maps-mcp-server` 实现地图与路线规划能力
- 接入 `open-meteo-mcp` 实现实时天气查询能力
- 通过 MCP ToolProvider 将外部工具注册到 Agent，让模型能够在问答过程中自动选择并调用地图 / 天气工具

### 6. 负责流式对话能力实现

- 使用 WebFlux + SSE 改造聊天接口，实现后端流式返回
- 基于 LangChain4j Reactor 的流式适配能力输出 `Flux<String>`，优化长回复场景下的交互体验
- 前端通过 Fetch Stream API 实现边接收边渲染，解决等待全部生成后再显示的问题

### 7. 负责前端聊天界面开发

- 基于 Vue 3 + Vite 开发单页聊天界面，采用 ChatGPT 风格布局
- 实现回车发送、自动滚动、AI 思考中状态、请求失败提示、流式消息展示
- 完成前后端联调，使聊天页面可直接驱动后端 Agent 服务

### 8. 负责数据库与初始化数据设计

- 设计 `schedule`、`reminder`、`course_material` 三张核心业务表
- 编写建表 SQL 与初始化数据脚本，支撑课表、提醒与资料检索功能
- 使用 MyBatis-Plus 实现实体类、Mapper 层与基础 CRUD 能力

## 项目亮点

### 1. Agent + Tool + RAG + MCP 融合架构

项目不仅实现了大模型问答，还将本地业务 Tool、课程资料 RAG 检索和外部 MCP 工具能力整合到统一 Agent 决策链路中，具备较强的可扩展性与工程完整度。

### 2. 面向真实业务场景的 Agent 设计

不是单纯做通用聊天，而是围绕校园场景抽象出课表、提醒、资料、天气、路线、出行建议等高频任务，使 Agent 具备明确业务边界和可验证的落地价值。

### 3. Skill 化规则设计

将“出行准备建议”抽象为结构化 Skill 规则，并约束 Agent 必须先查课表、再查天气、最后按课程 / 天气 / 温度 / 时间规则生成建议，体现了 Prompt 工程与业务规则结合的能力。

### 4. 流式对话体验完整落地

完成后端 WebFlux SSE 流式输出与前端流式渲染联调，使 Agent 对话体验更接近真实 AI 产品，而不是普通阻塞式接口返回。

### 5. 兼顾工程设计与业务可演示性

项目同时具备清晰的分层结构、数据库建模、配置化管理、异常处理、日志输出和前端可视化页面，适合作为 Agent 开发实习 / 大模型应用开发方向的完整项目案例。

## 面试展开版

适合在面试中按“技术挑战 - 解决方案 - 结果”进行展开表达。

### 1. 如何让 Agent 不只是普通聊天，而是真正可调用业务能力

技术挑战：

- 如果只做 Prompt 问答，模型容易编造课表、天气或提醒结果，无法满足真实业务场景

解决方案：

- 基于 LangChain4j 设计校园场景 Agent，将课表、提醒、课程资料、天气、路线等能力拆分为 Tool、RAG、MCP 三类能力源
- 通过系统提示词约束 Agent 的决策顺序，例如资料类问题优先检索 RAG，天气和路线类问题优先调用 MCP 工具
- 将出行准备建议抽象为规则化 Skill，要求 Agent 必须先查课表、再查天气、最后给出建议

结果：

- Agent 不再只是“回答问题”，而是具备了面向校园场景的任务执行能力
- 项目整体更贴近真实业务落地，而不是简单的大模型问答 Demo

### 2. 如何实现课程资料的检索增强问答

技术挑战：

- 课程试卷、复习资料、实验文档等内容较长，无法直接依赖模型上下文处理

解决方案：

- 设计课程资料知识库，将数据库中的资料内容加载后进行文档切分、Embedding 和向量存储
- 基于 RedisSearch 构建 RAG 检索链路，结合 LangChain4j 的 `EmbeddingStoreIngestor` 和 `ContentRetriever` 实现相似度检索
- 当用户提问课程资料相关问题时，由 Agent 自动选择知识库检索而非自由生成

结果：

- 能稳定支持“数据库期末试卷有哪些”“操作系统实验报告模板是什么”这类检索型问题
- 提升了答案的相关性和可控性

### 3. 如何接入地图和天气等外部实时能力

技术挑战：

- 天气和路线属于实时外部信息，不能通过本地写死逻辑或静态知识库处理

解决方案：

- 引入 MCP 机制，将高德地图 MCP Server 与 Open-Meteo MCP Server 接入后端
- 通过 `ToolProvider` 将外部工具注册到 Agent，使其能够像调用本地 Tool 一样调用外部能力
- 在 Prompt 中明确天气和路线问题的优先调用策略，避免模型绕过工具直接猜测

结果：

- 项目具备了实时天气查询和路线规划能力
- 架构上也为后续接入更多 MCP 工具留出了标准扩展路径

### 4. 如何实现更好的 AI 对话交互体验

技术挑战：

- 普通同步接口在长回复或外部工具调用场景下体验较差，用户需要等待完整结果返回

解决方案：

- 后端使用 WebFlux + SSE 实现流式输出
- 流式 Agent 使用 LangChain4j 的流式能力返回 `Flux<String>`
- 前端使用 Fetch Stream API 按分片读取并实时渲染消息内容

结果：

- 用户能够看到 AI 回复逐步生成，而不是等待整段返回
- 项目整体交互体验更接近真实 AI 产品
